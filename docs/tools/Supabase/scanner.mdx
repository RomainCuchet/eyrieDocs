## Overview

The scanner targets Supabase Data API endpoints and supports two operating modes:

1. Schema-driven scan (automatic path discovery from OpenAPI)
2. Path-driven scan (manual endpoints via `--paths`)

It is designed to work with JWT keys and modern Supabase key formats (`sb_publishable_*`, `sb_secret_*`).

## Input Model

- `--base-url` is used exactly as provided.
- `--api-key` is required.
- `--paths` is optional and enables manual path scanning.

No automatic URL rewriting is performed.

## Authentication Behavior

The scanner always sends:

- `apikey: <api-key>`

It conditionally sends:

- `Authorization: Bearer <api-key>` only if the key looks like a JWT.

This avoids invalid Bearer usage with `sb_publishable_*` and `sb_secret_*` opaque tokens.

## Scan Flow

1. Request schema from `<base-url>/`.
2. If schema is accessible, enumerate `paths` and test each endpoint.
3. If schema is not accessible:
- use manual fallback if `--paths` is set
- otherwise stop with a critical error and usage hint
4. For each endpoint:
- try `GET` first
- add `select=*` for non-RPC endpoints
- always add `limit=1` to GET requests unless `--evil` is set
- optionally fallback to `POST` for RPC/post-capable routes if `--agresive` is set

## Behavioral Flags

- `--aggresive`
: Enables broader POST fallback for POST-capable endpoints.

- `--evil`
: Disables safe limiting and attempts larger/full payload reads.

- `--paths users,orders,/rpc/my_fn`
: Provides explicit fallback targets when schema discovery is blocked.

:::warning Legal Notice
`--evil` allows the scanner to attempt to retrieve *all records* from each table, not just the first entry. `--aggresive` may trigger unintended actions or side effects on the target system, especially when POST requests are sent to endpoints.<br/><br/>

## Warning and Error Behavior

When schema fetch fails and the base URL does not include `/rest/v1`, the scanner prints:

- warning about potential missing `/rest/v1`

If schema fetch fails and `--paths` is missing, it prints a critical error and exits.

## Output Format

Results are written under:

- `web/supabase/output/`

### Filename

The output filename prefix is built from the Supabase instance name extracted via regex from base URL:

- regex source: part between `//` and `.supabase`
- example: `https://mijkjflyivdlfsgfcyjw.supabase.co/rest/v1` -> `mijkjflyivdlfsgfcyjw`

When a JWT `ref` can also be decoded and differs from instance name, both may be included.

### JSON payload

Each output file contains:

- `instance`: inferred Supabase instance name from URL
- `schema`: OpenAPI schema object when available (or `{}`)
- `options_used`: execution metadata
- `results`: endpoint-level scan results

`options_used` includes:

- `used_flags`: names of all flags used in the run
- `api_key_type`: inferred key type
- `with_values`: only safe values (`--base-url`, `--paths` when provided)
- `without_values`: sensitive/boolean flags without values (for example `--api-key`, `--aggresive`, `--evil`)

## API Key Type Inference

The scanner infers and stores one of:

- `sb_publishable`
- `sb_secret`
- `jwt`
- `unknown`

Inference is prefix-based for `sb_*` keys and structure-based for JWT.

#### Advanced Usage

<Tabs groupId="shell">
  <TabItem value="powershell" label="PowerShell" default>

```powershell

py .\scanner.py --base-url "https://<project>.supabase.co/rest/v1" --api-key "..." --aggresive

# Full data extraction (use with caution)
py .\scanner.py --base-url "https://<project>.supabase.co/rest/v1" --api-key "..." --evil

# Combine flags and manual paths
py .\scanner.py --base-url "https://<project>.supabase.co/rest/v1" --api-key "..." --aggresive --evil --paths users,orders
```

## Demo

:::note
This demo is based on [JAMstack case study](../../pentest/FictionnalCaseStudy/JAMstack.mdx) and uses the same intentionally misconfigured Supabase instance.
:::

![Scanner Demo](/img/Tools/Supabase/Scanner/scanner_demo.png)

1. The scanner is lauched using a `sb_publishable` type key and the base URL without `/rest/v1` suffix. The scanner detects the missing REST prefix and continues with a warning.
2. The scanner fails to fetch the OpenAPI schema (as it is blocked by the configuration) and prints a critical error with usage hints. Many misconfigured real-world instances enable public access to the schema, which would allow the scanner to automatically discover all endpoints without manual input.
3. The scanner is relaunched with the `--paths` flag to provide manual fallback endpoints. It successfully tests the `users` endpoint and retrieves the first user record, demonstrating the potential data exposure from the misconfiguration.
4. The scanner is relauched with the `sb_secret` type key and the same base URL. It is just for demonstration purposes as `sb_secret` keys are not meant to be used in frontend code and should not be exposed. The result is the same as if OpenAPI schema was exposed as described in step 2.

As `rcp/inc_profit is exposed` anyone that's able to discover it can modify the marketplace profit without any authorization.
